PROJECT GUIDE
In-Person Attendance Tracking

This file is safe to commit to GitHub. It intentionally contains no secrets,
real environment values, database URLs, JWT secrets, API keys, or passwords.


1. PROJECT OVERVIEW

This project is a full-stack in-person training attendance application.
Admins create training sessions, display or share a QR attendance link, collect
participant attendance, optionally upload nomination lists, and export final
attendance reports.

Main users:

- MASTER_ADMIN:
  Can see all trainings and manage admin user approvals.

- ADMIN:
  Can create and manage only the trainings they created.

- Participants:
  Do not log in. They scan/open a public attendance link and submit Employee ID
  and Employee Name while the attendance window is active.

Core flow:

1. Admin signs up or logs in.
2. MASTER_ADMIN approves pending admin users.
3. Admin creates a training with name, trainer, location, time window, and
   optional description.
4. App generates a QR/public attendance link for the training.
5. Participants scan the QR link and submit attendance.
6. Admin can upload nominations for expected attendees.
7. Admin can view live attendance, stop attendance manually, export Excel, or
   delete a training.


2. FOLDER STRUCTURE

client/
Frontend React application built with Vite. Contains all UI pages, reusable
components, API client code, CSS, and frontend deployment config.

server/
Backend Node.js/Express API. Contains routes, auth, Prisma access, QR
generation, Excel import/export, and backend deployment config.

server/prisma/
Prisma schema, migrations, and seed script. Edit this area when changing the
database model or creating seed data. Do not put environment values here.

client/src/pages/
Top-level routed React pages. Most user-facing screens live here.

client/src/components/
Reusable React UI pieces such as header, protected route wrapper, and backend
wake/status banner.

client/src/services/
Frontend API client and request helpers. This is where browser requests to the
backend are centralized.

client/src/utils/
Shared frontend utility functions. Session status and time-window display logic
live here.

6. BACKEND EXPLANATION

server/src/index.js is the main backend file. It initializes Express, Prisma,
CORS, JSON parsing, auth middleware, route handlers, and global error handling.

Auth routes:
- POST /api/auth/signup
- POST /api/auth/login
- GET /api/auth/me
- POST /api/auth/forgot-password
- POST /api/auth/reset-password

Training routes:
- POST /api/trainings
- GET /api/trainings
- GET /api/trainings/:id
- GET /api/trainings/:id/qr
- PATCH /api/trainings/:id/stop
- DELETE /api/trainings/:id

Attendance routes:
- GET /api/attend/:token/status
- POST /api/attend/:token
- GET /api/trainings/:id/attendance

Nomination routes:
- GET /api/trainings/:id/nominations
- POST /api/trainings/:id/nominations

Export routes:
- GET /api/trainings/:id/export

Ownership checks:
- MASTER_ADMIN can access all trainings.
- ADMIN can access only trainings where createdById matches their user ID.
- getTrainingOrThrow and requireTrainingAccess enforce ownership for protected
  training operations.

CORS:
- Configured near the top of server/src/index.js.
- Allows configured client URL, localhost dev URLs, and Vercel app origins.

Error handling:
- Global 404 handler returns Route not found.
- Global error handler converts known Multer and Prisma errors into friendly
  JSON responses.
- Unknown server errors return a generic message and are logged server-side.

Request ID logging:
- Dedicated request-ID middleware is not currently implemented.
- Current backend logging is limited to startup logging and server-side error
  logging.
- If request IDs are needed later, add middleware near the Express setup in
  server/src/index.js and include the ID in response headers and controlled
  server logs.


7. DATABASE / PRISMA MODELS

User

Represents an admin or master admin user.

Important fields:
- id
- username
- passwordHash
- role
- status
- resetToken
- resetTokenExpires
- createdAt
- updatedAt

Roles:
- MASTER_ADMIN
- ADMIN

Statuses:
- PENDING
- APPROVED
- REJECTED


Training

Represents a training session.

Important fields:
- id
- trainingName
- trainerName
- location
- description
- startDateTime
- endDateTime
- token
- manuallyStopped
- stoppedAt
- createdById
- attendances
- nominations

Rules:
- token is unique and powers the public attendance link.
- createdById controls admin ownership.


Attendance

Represents one attendance submission.

Important fields:
- id
- trainingId
- employeeId
- employeeName
- ipAddress
- submittedAt

Rules:
- trainingId + employeeId is unique.
- Employee ID must be exactly 10 digits.
- The public attendance route also checks trainingId + ipAddress as a
  best-effort one-submission-per-device/IP control.


Nomination

Represents one nominated/expected attendee for a training.

Important fields:
- id
- trainingId
- employeeId
- employeeName
- createdAt
- updatedAt

Rules:
- trainingId + employeeId is unique.
- Used to compare expected vs present attendees in details/export flows.


Important application rules:

- MASTER_ADMIN sees all trainings.
- ADMIN sees only trainings they created.
- Public attendance route does not require login.
- Employee ID must be exactly 10 digits.
- One submission per training per employee ID is enforced by Prisma/database
  uniqueness.
- One submission per training per IP/device is best effort and not foolproof.
  VPNs, shared networks, mobile carriers, and proxy behavior can affect it.

9. DEPLOYMENT NOTES

Frontend on Vercel:

- Root/build directory: client
- Install command: npm install
- Build command: npm run build
- Output directory: dist
- Set VITE_API_BASE_URL to the deployed backend API URL.

Backend on Render:

- Root/build directory: server
- Install command: npm install
- Build command: npm run build
- Start command: npm start
- Set DATABASE_URL, JWT_SECRET, CLIENT_URL, and PUBLIC_BASE_URL in Render
  environment variables.
- Run Prisma migrations during deployment using the configured Prisma deploy
  script when appropriate.

Database on Supabase:

- Use Supabase Postgres as the DATABASE_URL target.
- Apply Prisma migrations before relying on new schema fields.
- Keep database credentials only in environment variables.

Do not commit:
- .env
- *.env
- node_modules
- dist/build artifacts
- logs

How to add a new page:

1. Create a new file in client/src/pages/.
2. Add a route in client/src/App.jsx.
3. Add navigation in Header.jsx if it is a logged-in page.
4. Add API helpers in client/src/services/api.js if the page calls the backend.
5. Add backend routes in server/src/index.js if new API behavior is needed.


How to update API URL:

- For frontend deployment, set VITE_API_BASE_URL in the hosting provider.
- For local development, use a local frontend .env file that is not committed.
- API fallback is defined in client/src/services/api.js.


How to reset seed users:

- Review server/prisma/seed.js.
- Do not commit real production passwords.
- For production, prefer environment-driven seed passwords or manual admin
  creation with secure credentials.
- Run the Prisma seed command only against the intended database.


11. SECURITY NOTES

- Do not expose passwords in code, docs, logs, screenshots, or commits.
- Do not commit .env or environment-specific config values.
- Reset all demo/default passwords before any real production use.
- Password reset token shown in the UI is demo-mode behavior.
- Production should send password reset links by email and avoid displaying raw
  reset tokens on screen.
- Keep JWT_SECRET long, random, and private.
- Keep DATABASE_URL private.
- Public attendance links do not require login by design, so keep validation and
  duplicate checks on the backend.
- The IP/device duplicate block is best effort only and should not be treated as
  strong identity verification.
